Hands-On Python

Teach the core ideas of Python with LEGO bricks, paper cups, dice and movement — before anyone touches a keyboard. Every activity ends with a "bridge to code" so students see the real Python their hands just acted out.

🤚 Learn by Doing 🧩 6 Core Concepts 💻 No Computers Needed 👥 Pairs & Small Groups ⏱️ ~50 min Each

🧠 Why Hands First?
The idea is the hard part — not the syntax

Beginners stall on Python because abstract ideas (a variable, a loop, a function) have nothing to grab onto. If a student has already physically been a loop — walked it, felt it repeat — the keyword for is just a label for something they already understand.

Every activity ends at real code

This is not "computing without computers" for its own sake. Each lesson finishes with a Bridge to Python: the exact code that matches what their hands just did. Move to the keyboard the moment the concept has landed.

Run the lessons in order — each one reuses the physical "vocabulary" of the last. By Lesson 6 a function is simply "a machine that does a sequence, with cups for inputs and a loop inside".

Lesson 1 · Sequencing & Algorithms
Be the Robot 🤖
Step-by-step instructions · Order matters · Algorithms · The first Python program
🤖
▼
By the end of Lesson 1 students will understand that a program is an ordered sequence of instructions, that a computer follows them literally, and that changing the order changes the result. They'll write and read their first lines of Python.
⏱️ 50 min 👥 Pairs 🧱 LEGO + index cards
🎒 Kit for this lesson
  • 6–8 LEGO bricks per pair
  • 1 baseplate per pair
  • Blank index cards
  • A pen each
Part 1 — The Build, No Talking
  • One partner builds in secretBehind a book or bag, Partner A builds a small 4–6 brick model on their baseplate. Nobody else sees it.
  • Write the instructions on cardsPartner A writes one instruction per index card to recreate the model — e.g. "Place the red 2×4 brick flat in the bottom-left corner". One action per card, no diagrams.
  • Partner B becomes the robotPartner B takes the cards in the given order and follows each one exactly — no guessing, no improving. A robot does only what it is told.
  • CompareReveal both models. Where they differ, the instruction was unclear or in the wrong place — not the robot's fault. This is debugging.
🧠 Key Concept: An algorithm is an ordered list of steps

A program runs top to bottom, one line at a time. The computer never skips ahead and never fills in gaps. The stack of cards is a program — and the order of the cards is the program's logic.

Part 2 — Order Matters
  • Shuffle two cardsSwap the order of any two instruction cards and run the robot again. Watch the build break — "put the roof on" before "build the walls" simply cannot work.
  • Find the smallest fixChallenge the pair to fix the build by moving just one card. This is their first taste of reordering code to fix a bug.
Keep a "robot rule" poster up all year: The robot does exactly what the card says, in the order the cards are in. You'll point at it constantly.
🌉 Bridge to Python

Each index card is one line of code. Reading top to bottom is exactly how Python runs a file.

# A program is a sequence — steps run top to bottom, in order. print("Place the red brick") print("Place the blue brick on top") print("Place the green brick on top") print("Tower built!")
Swap two print lines and run it — the output order changes, just like the shuffled cards. The computer never objects; it just obeys.
🔥 Extension Challenge

Give a pair a stack of cards with one bug card hidden inside (e.g. a step that's impossible or out of place). Can they act as robot, spot exactly which line fails, and rewrite that one card? That's debugging — and it's a skill, not a punishment.

✅ Lesson 1 Checklist
  • Each pair built a model from written instructions alone
  • Students saw a build break when two steps were swapped
  • The class can state the "robot rule" in their own words
  • Everyone has read (and ideally typed) a 3–4 line Python program
  • Students can explain: what an algorithm is, and why order matters

Lesson 2 · Variables & Assignment
The Labelled Cup 🥤
Named containers · Assignment · Reassignment · Data types
🥤
▼
By the end of Lesson 2 students will understand a variable as a labelled container that holds one value at a time, that assigning a new value replaces the old one, and that values have different types (numbers, words, true/false).
⏱️ 50 min 👥 Pairs 🥤 Cups + sticky notes + bricks
🎒 Kit for this lesson
  • 3 paper cups per pair
  • Sticky notes for labels
  • LEGO bricks (the "values")
  • A few small slips of paper for words
Part 1 — Make a Variable
  • Label a cupStudents stick a label on a cup that reads score. The label is the variable's name; the cup is the storage.
  • Put a value inDrop a brick with "0" written on it into the score cup. The cup now holds the value 0.
  • Make two moreLabel a second cup lives and put a "3" brick in it. Label a third cup player and put a paper slip reading "Sam" in it.
🧠 Key Concept: name = value

A variable has two parts: a name (the label) and a value (what's inside). In Python the = sign means "put this value into this cup" — it is not the equals from maths.

Part 2 — Reassignment
  • Score a pointTell students: "the player caught a fish — score is now 10". They must take the 0 brick out and put a "10" brick in. A cup holds only one value at a time.
  • Where did the 0 go?Ask the class. The old value is gone — overwritten. This is the single most important idea in the lesson.
  • Change a life"The player got hit." Swap the "3" brick in lives for a "2". Same cup, same label, new value.
Watch for the maths misconception: students think score = 10 is a fact that's "true forever". Keep physically emptying the cup — the value is whatever is in there right now.
Part 3 — Types of Value

Sort the cup contents into kinds: number bricks (you can add them), word slips (you can't add them), and a "true/false" coin. These are Python's data types: int, str, bool.

🌉 Bridge to Python
# Each line: put a value into a labelled cup. score = 0 # a number (int) score = 10 # the old 0 is gone — replaced lives = 3 player = "Sam" # a word (str) print(player, "has", score, "points and", lives, "lives")
Run it, then add the line score = score + 5. Read it as: "take what's in the score cup, add 5, put the result back in the score cup." Have students mime it with the brick.
🔥 Extension Challenge

The classic swap puzzle, with cups. Give pairs a left cup and a right cup with different bricks. Challenge: get them to swap contents. They'll discover they need a third empty cup — temp. That's a real technique professional programmers use.

✅ Lesson 2 Checklist
  • Each pair built and labelled three variable cups
  • Students physically reassigned a variable and saw the old value disappear
  • The class can explain why = is not the maths "equals"
  • Students sorted values into number / word / true-false types
  • Everyone has typed a Python program that creates and updates variables

Lesson 3 · Lists & Indexing
The LEGO Brick Train 🚂
Ordered collections · Index starts at 0 · Append & remove · Length
🚂
▼
By the end of Lesson 3 students will understand a list as an ordered collection of values, that each item has a numbered position (index) starting at 0, and that lists can grow and shrink.
⏱️ 50 min 👥 Pairs 🧱 Bricks + masking tape
🎒 Kit for this lesson
  • 6+ LEGO bricks of different colours per pair
  • Masking tape to mark index numbers
  • A long strip of desk or floor
Part 1 — Build the Train
  • Lay the trackEach pair puts down a strip of masking tape and writes index numbers along it: 0, 1, 2, 3, 4… Start at 0 — this is non-negotiable and the whole point.
  • Park the bricksPlace 4 coloured bricks on positions 0–3. This is the list train = ["red", "blue", "green", "yellow"].
  • Fetch by indexCall out an index — "give me index 2!" — students physically pick up the brick at position 2. Repeat fast, like a game.
🧠 Key Concept: Index starts at 0

The first item is at index 0, not 1. The brick at index 2 is the third brick. This trips up everyone at first — which is exactly why we tape the numbers down and point at them.

Part 2 — Grow and Shrink
  • Append"Add an orange brick to the end." Students place it at the next free index. The list got longer — that's .append().
  • Remove"Take out the blue brick." Remove it — and notice everything after it shuffles down a position. Indexes are not permanent name tags; they are current positions.
  • Count"How long is the train?" They count the bricks. That's len(train).
When a brick is removed and the rest shuffle down, that's the best moment of the lesson. Pause on it. "Index 3 used to be yellow — what is it now?"
🌉 Bridge to Python
# The train of bricks, written as a Python list. train = ["red", "blue", "green", "yellow"] print(train[0]) # first brick -> red print(train[2]) # third brick -> green train.append("orange") # add a brick to the end print(len(train)) # how many bricks -> 5
Ask for train[10] on the real train — there's no brick there. Run it in Python: IndexError. The error message is the computer saying "there's nothing at that position."
🔥 Extension Challenge

Give a pair a target pattern (e.g. red-red-blue-green) and a scrambled train. They must reach the target using only "swap index A and index B" moves, said out loud before each swap. They're now writing list-manipulation algorithms — and they'll meet sorting later.

✅ Lesson 3 Checklist
  • Each pair built an indexed brick train starting at position 0
  • Students can fetch the correct brick when given an index
  • The class saw indexes shift when an item was removed
  • Students used append and len on their physical train
  • Everyone has typed Python that indexes and grows a list

Lesson 4 · Loops & Repetition
Walk the Loop 🔁
for loops · while loops · The loop body · Indentation = "inside the box"
🔁
▼
By the end of Lesson 4 students will understand a loop as repeating a set of actions, the difference between repeating for each item (for) and repeating until something changes (while), and that the indented body is what gets repeated.
⏱️ 50 min 👥 Small groups 🧍 Floor space + brick train
🎒 Kit for this lesson
  • The brick train from Lesson 3
  • Open floor space
  • Masking tape to mark a walking circuit
  • A baseplate as the "build target"
Part 1 — The for Loop: Once Per Brick
  • Line up the trainPut the brick train on the table. The instruction is: "for each brick in the train, place a matching brick on the baseplate."
  • Walk itA student touches brick 0, does the action, moves to brick 1, does the action… The loop runs exactly once per item and then stops on its own. No item left = loop over.
  • Name the partsThe thing being walked is the list. The current brick is the loop variable. The action performed is the loop body.
Part 2 — The while Loop: Until It Changes
  • Tape a circuitMark a small walking loop on the floor. The rule: "while you have taken fewer than 5 steps, take another step."
  • Walk and checkBefore every step the student says the count and checks the condition out loud: "2 steps, fewer than 5? Yes — step again." When the check fails, they stop.
  • Break it on purposeRemove the "+1 step" — now the count never changes, the check is always true, and the student would walk forever. That's an infinite loop, felt in the legs.
🧠 Key Concept: Indentation is "inside the loop"

In Python the repeated actions are indented underneath the loop line. Indented = inside the box that repeats. Not indented = outside, runs once. Have students physically step in and out of a taped box to feel it.

🌉 Bridge to Python
for — once per item
train = ["red", "blue", "green", "yellow"] for colour in train: print("Place a", colour, "brick") # indented = the loop body
while — until the condition changes
steps = 0 while steps < 5: print("Step", steps) steps = steps + 1 # this is what eventually stops the loop print("Done walking.") # not indented — runs once, after the loop
Delete the steps = steps + 1 line and run it (be ready to stop it with the keyboard interrupt). The class just walked this bug — they'll never forget what causes an infinite loop.
🔥 Extension Challenge

Nested loops, with bodies. "For each row (1 to 3), for each brick in the train, place one brick." One student walks the outer loop, a partner walks the inner loop and resets each time. They've physically performed a nested loop — the thing that builds grids, levels and game boards.

✅ Lesson 4 Checklist
  • Students walked a for loop, doing the body once per item
  • Students walked a while loop, checking the condition before each pass
  • The class created — and felt — an infinite loop, then fixed it
  • Students can point to the loop body and explain why it's indented
  • Everyone has typed and run both a for loop and a while loop

Lesson 5 · Conditionals & Decisions
The Decision Floor 🎲
if / elif / else · Conditions · Comparison · Only one path runs
🎲
▼
By the end of Lesson 5 students will understand that a conditional chooses a path based on whether something is true, that if / elif / else are checked in order, and that exactly one path runs.
⏱️ 50 min 👥 Whole class + pairs 🎲 Tape + dice
🎒 Kit for this lesson
  • Masking tape to mark a branching path on the floor
  • One die per group
  • Signs: "FORWARD 3", "FORWARD 1", "STAY"
Part 1 — Build the Decision Floor
  • Tape a forkLay a path that splits into three branches. Put a sign at the start of each: roll == 6, roll >= 4, and else.
  • Roll the conditionA student rolls the die at the junction. The number is the value being tested.
  • Walk one branch onlyThey check the signs top to bottom: is it 6? If not, is it 4+? If not, take the else branch. They walk one path — and the other two are not touched.
🧠 Key Concept: Checked in order, one winner

Python tests if first, then each elif in order, and stops at the first true one. else is the "none of the above" path. A roll of 6 is also "4 or more" — but because if is checked first, the 6 branch wins. Order of the signs matters.

Part 2 — Conditions Are True or False
  • Say the condition out loudBefore stepping, the walker states it as a yes/no question: "Is the roll equal to 6? No. Is the roll 4 or more? Yes." Every condition collapses to true or false.
  • Reorder and break itSwap the roll >= 4 sign above the roll == 6 sign. Now a 6 never reaches its own branch — it's caught early. Students feel why condition order is a real bug.
Build a wall of comparison cards: == equals, != not equal, > greater than, >= at least. Students hold up the card that matches each spoken question.
🌉 Bridge to Python
# The decision floor, written as a conditional. roll = 4 if roll == 6: print("Move forward 3 spaces") elif roll >= 4: print("Move forward 1 space") else: print("Stay where you are")
Change roll to 6, 5, 3 and 1 and re-run each time. Each value walks a different branch — and only ever one. The indented lines under each branch are that path, exactly like the taped floor.
🔥 Extension Challenge

Combine with Lesson 4: put a conditional inside a loop. "For each roll in a list of 10 rolls, decide the move." One student walks the loop, handing each roll to a partner standing on the decision floor. That nested structure is the core of almost every game's logic.

✅ Lesson 5 Checklist
  • The class built a three-branch decision floor with conditions on signs
  • Students walked exactly one branch per roll, checking signs in order
  • Students saw a 6 get "caught early" when sign order was swapped
  • The class can read comparison operators as yes/no questions
  • Everyone has typed an if / elif / else and run it with different values

Lesson 6 · Functions & Parameters
The Brick Machine ⚙️
Reusable recipes · Parameters as inputs · Return values · Calling a function
⚙️
▼
By the end of Lesson 6 students will understand a function as a named, reusable recipe, that parameters are the inputs you hand it, and that a return value is what it hands back. This lesson ties together everything from Lessons 1–5.
⏱️ 50 min 👥 Groups of 3 ⚙️ Bricks + cups + baseplate
🎒 Kit for this lesson
  • LEGO bricks of several colours
  • 2 labelled cups per group (the parameters)
  • A baseplate (the build area)
  • An index card with the recipe written on it
Part 1 — Become the Machine

Groups of three take on a role each. The machine never decides anything — it just follows its recipe card with whatever inputs it's handed.

⚙️ The Machine
Holds the recipe card and follows it exactly. Takes whatever is in the input cups, builds, and hands back a result.
📥 The Caller
Fills the input cups with chosen values, says "go", and waits for the machine to return something.
👀 The Checker
Confirms the machine only used the cups it was given — not bricks from elsewhere — and notes what was returned.
Part 2 — Run the Recipe
  • Write the recipeOn the card: "build_wall(colour, height): place a [colour] brick for each row from 0 up to height, then return the words 'wall finished'." It reuses a loop from Lesson 4.
  • Call itThe Caller puts a red brick in the colour cup and a "3" brick in the height cup, then says "build_wall, go!"
  • Return a valueThe Machine builds, then hands back a slip reading "wall finished". That handed-back slip is the return value — the Caller can use it or store it in a cup.
  • Call it again — differentlyNow blue, height 5. Same machine, same recipe, new inputs, new result. That reuse is the entire point of a function.
🧠 Key Concept: Define once, call many times

A function is written once (the recipe card) and used many times with different inputs. The parameters are cups waiting to be filled by whoever calls it. return is the machine handing its result back out.

🌉 Bridge to Python
# Define the machine once... def build_wall(colour, height): for row in range(height): print("Place a", colour, "brick — row", row) return "wall finished" # ...then call it as many times as you like. result = build_wall("red", 3) # hand the machine two inputs print(result) # -> wall finished build_wall("blue", 5) # same recipe, new inputs
Map it back out loud: def is writing the recipe card, the names in brackets are the input cups, the indented lines are the recipe steps, and return is the slip handed back. Nothing here is new — it's Lessons 1–5 in a reusable box.
🔥 Extension Challenge

Have one machine call another machine. build_tower calls build_wall three times with different colours. Students physically pass cups between two machine-students. They've just discovered that programs are functions built out of functions — which is how every real piece of software is made.

✅ Lesson 6 Checklist
  • Each group ran a "machine" that followed a fixed recipe card
  • Students filled parameter cups and saw the same function reused with new inputs
  • The class can explain what a return value is and where it goes
  • Students connected def / parameters / return to the physical roles
  • Everyone has typed, called, and reused a Python function

Your students have now physically performed sequencing, variables, lists, loops, conditionals and functions — and seen the exact Python for each. Those six ideas are the whole foundation. Everything else in programming is a combination of things their hands already understand.
